Qué está hecho, qué queda

Tres columnas: lo que ya funciona end-to-end (verde), lo que está parcial pero levantable (ámbar), y lo que es la próxima capa de producto (rojo = pendiente, no urgente — nada bloquea la demo).

Hecho · funcional

  • ✓Auth product-grade: signup con Pulso ID, login, refresh persistido, logout, change password, MFA TOTP, lockout 5×15min
  • ✓Verificación email + recuperación contraseña con tokens de un solo uso
  • ✓RBAC con 6 roles + decorators @Roles/@CurrentUser/@Public
  • ✓QR de emergencia: crear/revocar/listar logs, página pública, imprimible A4
  • ✓Portal profesional: búsqueda DNI, consent flow, timeline, carga evolución
  • ✓Mica IA: chat con Anthropic + mock seguro fallback + guardrails anti-prescripción
  • ✓Audit log con hash chain SHA-256 + triggers SQL append-only
  • ✓Storage real con MinIO + presigned URLs + guard owner-or-consent
  • ✓Notificaciones SMTP via nodemailer + persistencia + multi-canal
  • ✓Recetas digitales con signatureHash + dispensación por farmacia
  • ✓Recordatorios de medicación con presets de frecuencia
  • ✓Admin operativo: cambio rol + suspender + audit verify chain
  • ✓Pino structured logs con redacción de PII médica
  • ✓Health endpoints: liveness + readiness con DB + chain check
  • ✓Mobile Expo SDK 51 con 4 pantallas (ID, QR, Mica, Docs)

Parcial · funciona pero falta robustez

  • ○UI de sesiones activas en /panel/seguridad (placeholder; falta endpoint /auth/sessions)
  • ○Recordatorios persistidos pero sin cron job que dispare a la hora (modelo tiene lastFiredAt listo)
  • ○WhatsApp y Push: dispatcher persiste pero canales no enviados realmente (solo EMAIL e IN_APP)
  • ○Verificación de matrícula: campo libre en seed; no hay integración con colegios
  • ○Verificación de DNI en signup: solo formato, no contra RENAPER
  • ○Consent flow: en MVP el profesional crea consent vigente directo (modo demo); en producción requiere confirmación push del ciudadano
  • ○Tests: unitarios en auth + audit (15 tests). Falta integration y E2E
  • ○Pulso Connect: stub con README + estructura, sin conectores reales HL7/FHIR

Cómo seguir · 3 caminos

Tres opciones priorizadas según el objetivo. No son excluyentes — se pueden hacer en serie. Mi recomendación: arrancar por el camino A (operativo) que es el más corto y deja Pulso 100% autónomo.

⚡ A · Esta sesión

Operativo end-to-end

Cierra los gaps que dejan a Pulso depender de intervención manual. Lo que hace falta para piloto.

  • Cron job recordatorios (@nestjs/schedule)
  • Endpoint /auth/sessions con revocar individual
  • UI sessions activas conectada
  • Verificación email reenvío desde panel
  • Consent flow con aceptación del ciudadano (no auto-grant)
🛰 B · 1-2 sesiones

Canales reales y deploy

Producción real con WhatsApp, push y deploy a la nube.

  • Twilio WhatsApp Business + templates aprobados
  • Expo Push en mobile + EAS Build
  • Deploy api a Render + web/admin a Vercel
  • Supabase managed Postgres + Cloudflare R2 storage
  • CI/CD con preview deploys por PR
🏛 C · Mediano plazo

Interoperabilidad y compliance

Lo que hace a Pulso defendible y vendible a gobiernos.

  • Conector HL7v2 + FHIR R4 (primer hospital piloto)
  • OAuth 2.0 server para terceras apps
  • RENAPER para verificación DNI
  • Verificación matrícula vs colegios profesionales
  • DPIA + auditoría externa pre-piloto
  • Tests E2E + integration con testcontainers
